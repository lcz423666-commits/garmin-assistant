### 1.2 各核心文件职责与函数清单

#### `app/garmin_monitor.py`（主监控进程）
**职责**：守护进程入口，每轮轮询所有启用用户，拉 Garmin 数据→判断新活动/睡眠→调 LLM→推送微信，同时驱动 ICU 骑行和睡眠分析（仅丛至）。

主要函数：
- `load_enabled_users()` — 从 users.json 读取 enabled 用户
- `sync_runtime_users()` — 动态同步用户缓存
- `load_state/save_state()` — 每用户状态文件读写
- `login_garmin()` — 使用 token 目录或密码登录 Garmin
- `push_to_wechat()` — PushPlus 微信推送（含图表附件）
- `current_check_strategy()` — 判断是否高频时段（05-09h 每5分钟，其余每20分钟）
- `get_feishu_token/feishu_write_push_record/feishu_write_run_log()` — 飞书多维表格日志
- `feishu_enabled()` — 飞书是否启用
- `update_sleep_health_state/update_evening_push_state()` — 状态文件更新
- `bj_now()` — 北京时间工具
- `build_pushplus_payload()` — 构建推送 HTML

依赖：`garminconnect`（Garmin SDK）、`requests`（HTTP）、`openai`（LLM via SiliconFlow）

---

#### `app/phase1_builder.py`（数据构建层，120 KB，最大文件）
**职责**：构建所有 LLM payload（睡眠、运动、周报、初始报告、冷启动快照、基线），是分析逻辑的核心。

主要函数（部分）：
- `build_sleep_payload()` — 构建睡眠分析 payload（含 SPO2 告警、HRV、血氧连续性检测）
- `build_activity_payload()` — 构建运动分析 payload（含骑行专项、左右平衡、踏频、TSS/IF）
- `build_weekly_payload()` — 周报 payload
- `build_initial_7d_summary_payload()` — 新用户接入后7天初步分析
- `build_initial_summary_payload()` — 30天综合分析
- `build_cold_start_snapshot()` — 冷启动快照
- `build_user_baseline_payload()` — 30天基线刷新
- `_sleep_spo2_alert_context()` — SPO2 连续告警检测（单晚/多晚/持续）
- `_sleep_forced_alerts()` — 强制告警生成
- `_activity_forced_alerts()` — 运动强制告警
- `summarize_cycling_specific()` — 骑行专项摘要（功率区间、踏频、前中后段分析）
- `build_performance_highlights()` — VO2Max/FTP 提升亮点
- `derive_weekly_intensity_minutes()` — WHO 每周强度分钟数

### 2.2 System Prompts 原文

#### SLEEP_PROMPT（睡眠晨报）

```
你是一个懂 Garmin 数据和训练恢复的微信陪伴型教练。

你的任务不是重新做分析，而是把程序已经整理好的睡眠判断，写成一条有重点深度的睡眠晨报。

必须遵守：
- 目标长度 400-650 字，最多 650 字
- 优先直接使用 `preferred_opening` 开头；如果不是早上，不要硬写"下午好/晚上好"
- 如果 `preferred_opening` 已经给了称呼后的判断句，正文第一段必须直接沿用这句，不要只保留称呼后再自己改回泛泛开头
- 开头第一句必须同时包含总体判断和最关键问题，优先直接点名最低血氧、HRV、静息心率等具体异常，不要先铺一句模糊判断再卖关子
- 全文不要出现"有一个明确提醒不能忽略"这句话，也不要用"但有个提醒要说""先说一个问题"这类悬念式模板句
- 面向用户展示任何时长时都统一人话化：大于等于 60 分钟写成"X小时Y分钟"，小于 60 分钟写"X分钟"
- 结构顺序必须是：开头 -> 总判断 -> 原因解释 -> 亮点 -> 问题点/提醒 -> 跟问题点绑定的建议 -> 连续趋势一句话
- 至少写出 2-4 个关键依据，不能只说"恢复不错"或"保持稳定"
- 如果 `forced_alerts` 不为空，必须把每一条都写进正文，不能吞掉
- 如果有 `user_stage`，要优先按阶段来写，而不是默认自己已经很懂用户
- `user_stage=observation`：只讲这晚本身 + 很轻的初步判断，不要写"你平时""你最近一直""你的一贯水平"，也不要做强基线对比
- `user_stage=early`：可以少量参考最近7天或最近一周，但要带"初步判断""从目前这段时间看"这类口气，不要写得太满
- `user_stage=mature`：可以自然引用最近30天常态和"你平时"的表达，但 baseline 仍然只用于帮助判断，不要逐条复述
- `user_stage` 只能影响整体语气和判断口径，不能压掉明显异常的提醒力度
- 如果出现明显异常值，比如最低血氧明显偏低、HRV 明显低于平衡区间、静息心率明显高于常态，这些提醒仍然要保留足够强度
- 如果 `spo2_alert_context.guidance_level=observe`，最低血氧可以按偶发波动来写，建议以继续观察为主，但要明确这是偶发下探
- 如果 `spo2_alert_context.guidance_level=escalated`，必须明确写出这已经是反复出现的信号，可能和夜间呼吸受阻有关；建议里自然提示关注打鼾、憋醒/喘醒、白天困倦、晨起头痛，且不要把主要建议写成一句轻飘飘的"继续留意"
- 如果 `spo2_alert_context.template_mode=persistent_issue`，整条文案都不要再按"单晚异常模板"来写；开头第一句就要点名这是反复出现的问题，不要写得像第一次看到
- 如果 `spo2_alert_context.recent_occurrence_text` 或 `spo2_alert_context.consecutive_occurrence_text` 不为空，正文里要优先明确写出"最近一周已经第几次出现""已经连续第几晚出现"
- 如果开头已经把主问题和连续性说清楚，后文不要再换说法重复"不是偶发波动""需要按连续异常处理""需要重视"这类同一个判断；后文改写它意味着什么、以及下一步怎么做就够了
- 全文禁止 markdown，不要加粗，不要加小标题
- 更像微信消息，不要像健康报告
- 少一点讲课感，少用"我们来看""值得关注""建议你"这类教练腔
- 输出格式要求：不要在文案中包含分隔线（---）、字数统计（如"约370字"）、或任何元标注。文案结尾应该是一个完整的、有实际内容的句子，不要用单独一两个词收尾。
```

## 六、实验分支：ICU + 骑行深度分析 路线图

### Phase 1 — 里程段拐点检测 【✅ 已完成】

拉 activity 的 streams.json（types=watts,heartrate,cadence），在代码里用滑动窗口检测"故事点"——主要是三类：持续高功率段（>FTP 的爬坡或主动进攻）、功率心率脱钩点（心率持续攀升但功率在降）、明显的衰减点（后半程同样功率心率比前半程高一个区间）。算法用 5 分钟滑动窗口，挑出最显著的 2-3 个点，带上里程位置（从 distance 流取）喂给大模型。不动天气、不动历史数据。

**当前已接入高德 API 做里程位置读取。**

**完成证据**：
- `app/icu_turning_points.py`：`detect_high_power_segments()`、`detect_decoupling_points()`、`detect_power_decay()` 全部实现，含5分钟滑动窗口
- `app/icu_cycling.py`：调用上述三个检测函数，并调用高德逆地理编码 API（`AMAP_REGEOCODE_URL`）
- `app/icu_turning_points_validator.py`：算法验证器已存在
- `/root/.env` 中 `AMAP_API_KEY` 已配置
- `/root/.icu_cycling_state.json` 中 `last_activity_id=i140263972`，说明已实际推送过

### Phase 2 — 风向分析 【⬜ 未开始】

加 latlng 流到 streams 请求里，每 2 公里算一次航向（两个 GPS 点之间的方位角），同时调 Open-Meteo API 按时间地点查当时的风向风速，把航向和风向做向量差——0-45 度是顺风、135-180 度是逆风、中间是侧风。输出按里程段汇总，和 Phase 1 的拐点合并喂给大模型。

**当前状态**：`icu_cycling.py` 中无 latlng 流请求，无 Open-Meteo 调用，无航向计算逻辑。

### Phase 3 — 历史基线对比 【⬜ 未开始】

建本地 SQLite 存每次骑行的关键指标（date、distance、FTP、IF、NP、AP、decoupling、TSS、各里程段功率心率均值）。每次新活动进来先和过去同距离±20%、同强度±10% 的骑行做对比，算出差值，和前两阶段数据合并喂给大模型。

**当前状态**：无 SQLite 数据库文件，无对应建表/读写逻辑。
